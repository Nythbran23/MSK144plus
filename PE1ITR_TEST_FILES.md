# PE1ITR test files — protocol mismatch with WSJT-X 3.0

## Summary

The synthetic test WAV files (`MSK144_60ms_pulse.wav`, `MSK144_140ms_l2.wav`,
`MSK144_20ms30ms_train_s1.wav`, etc.) were generated by PE1ITR in
**January 2017** using **MSHV's MSK144 v1 protocol**, which **predates** the
WSJT-X 1.8 (2018) refactor that moved to the (128, 90) LDPC + 77-bit message
format Roger's v2 port targets.

These files cannot be decoded by our current decoder without implementing
the older v1 protocol as a second decoder pathway.

## Source

Source for the test files: PE1ITR blog post 2017-01-13:

> The MSK144 message is "K1JT PE1ITR R+10". The FSK441 and MSK144 messages are
> random chopped in the same way to a pulse like a meteor scatter ping. After
> that white noise is added so the signals are "in the noise". The pulse
> length chosen is 40ms, 60ms and 140ms. … I generated and decoded the files
> with MSHV software.

Generated with MSHV 2017, which at that time used MSK144 **v1** format.

## Protocol differences: v1 (2017) vs v2 (2018+)

| Aspect | v1 (PE1ITR test files) | v2 (WSJT-X 3.0 / our port) |
|---|---|---|
| Sync word | `s8ms = [0,1,0,0,1,1,1,0]` | `s8 = [0,1,1,1,0,0,1,0]` |
| Channel layout | 8 sync + 48 codeword + 8 sync + 80 codeword (=144 bits) | Same 144-bit layout, different codeword |
| Message length | 80 bits (10 bytes × 8) | 77 bits + 3 type bits |
| LDPC code | (128, 80) regular column-weight-3 | (128, 90) irregular |
| Generator matrix `g[48]` | 48 × 80-bit hex strings | Different (90 → 128) |
| Encoder | `encode_msk144` (in MSHV) | `encode_128_90` (WSJT-X) |
| Decoder | `bpdecode128_80` | `bpdecode128_90` |

References in MSHV:
- v1 sync: `/tmp/MSHV/src/HvMsPlayer/libsound/HvGenMsk/config_rpt_msk40.h` line 4:
  `static const char s8ms[8] = {0,1,0,0,1,1,1,0}; //ok 4  78 invers msk40 14s`
- v1 encoder: `genmesage_msk.cpp` line 312, function `encode_msk144(message, codeword)`
- v1 vs v2 branch in TX: `genmesage_msk.cpp` lines 845-867 (the `if (msk144_v1) ... else ...` logic)
- v1 vs v2 branch in RX: `decodermsk144.cpp` line 67-73 (the `if (ss_msk144ms) ... else ...` switch)

## Why our pipeline gets so close but doesn't decode

When I ran my `diag_compare` tool on the burst extracted from
`MSK144_60ms_pulse.wav`:

- Squared spectrum cleanly shows tones at 2000 Hz and 4000 Hz → fc=1500 Hz ✓
- MSK40 candidate detection finds 5 candidates with detmet up to 1918 ✓
- MSK40 sync correlator (using `S8R = [1,0,1,1,0,0,0,1]`) gives `xmax=5.63` ✓
- Demod produces `n_bad_sync=0` (all 8 sync bits match) ✓
- BP **does not converge** ✗

The fact that sync correlation is so strong is a coincidence: the v1 `s8ms`
and our v2 MSK40 `s8r` overlap enough in their I/Q assignment that we get a
respectable but not-quite-locked correlation. Sync error count of 0 is
because we're also looking at hard bits at the right positions in time, but
the actual bit pattern after the sync is completely different — it's MSK144
v1 codeword bits, not MSK40 (32, 16) codeword bits.

When we feed the demodulated 32 payload softbits into our (32, 16) LDPC
decoder, there's no valid codeword corresponding to that bit pattern, so BP
never converges.

## What this would take to fix

To decode the PE1ITR files we'd need to implement an **MSK144 v1 decoder**:

1. Port the (128, 80) LDPC code:
   - Generator matrix `g[48]` (48 hex strings, 20 chars / 80 bits each)
   - Parity-check matrix structure (column-weight 3, derived from `g`)
   - BP decoder for (128, 80)
   - `encode_msk144` for round-trip validation
2. Add a second `s8ms` sync word and matched filter (matched filter cb_v1
   built from `s8ms` instead of `s8`)
3. Add a v1 message unpacker:
   - Unpack 80 bits → 10 bytes
   - Decode the 10-byte format into call1/call2/grid (or report)
4. In `mskrtd_decode`, try v1 sync alongside v2 sync. If v1 sync succeeds,
   use the v1 demod + LDPC + unpacker path

This is roughly the same complexity as implementing MSK40 — another ~1200 lines
of porting from MSHV's C++. Not impossible but substantial.

## Recommendation: defer v1 decoder

The v1 protocol is **legacy** — current operators on-air use v2 exclusively.
PE1ITR's 2017 test files are an artifact of when the new protocol hadn't yet
landed. The decoder is designed for current real-world traffic.

For our current goals — building a Rust port of WSJT-X 3.0 MSK144 — the v1
files don't represent something we need to decode. We should:

1. **Keep the v2 port as-is** (113 tests passing, including 8/8 real-WAV
   regression on actual on-air recordings)
2. **Document that the synthetic PE1ITR files use v1 protocol** and won't
   decode with this codebase
3. **Possibly later**: implement v1 as a separate optional decoder pathway
   if there's a reason to support it (e.g., archival decoding of old MS
   recordings)

## Real-world recordings that DO decode

later, recorded with WSJT-X 1.8+ producing v2-format signals. These decode
correctly with our v2 port.
